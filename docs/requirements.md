## 1. 背景・目的・成功基準

日本の公的オープンデータAPIを、英語圏の開発者・AIエージェントがそのまま使える構造化データとしてApify Storeで販売するActorを5本公開する。

**成功基準（受入とは別のビジネスKPI）**: 各Actorが公開後3ヶ月で月$100粗利、ファミリー合算で公開後6ヶ月に月$200粗利。保守は各Actor月1時間以内。詳細閾値は§10。

## 2. スコープ／非スコープ／前提

**スコープ**: Actor 5本（EDINET開示 / 補助金 / 不動産取引価格 / 法人エンリッチ / 法令）と、その公開に必要な共通品質基盤（回帰テスト・監視・出典表記・課金）。

**非スコープ**: レインズ等の二次利用禁止データ源／規約上スクレイピング禁止のサイト（J-PlatPat等）／インボイスWeb-API（柱3アドオン専用。国税庁の承認条件が特殊なためActor化しない）／SaaS・ダッシュボード化／TDnet適時開示（権利未整理につき保留）／能動マーケ（SNS運用・広告）。

**前提（申請状況とゲート）**:

| データ源 | 認証 | 状態（2026-07-07） | 影響 |
|---|---|---|---|
| EDINET API v2 | APIキー | 取得済み | Actor#1は即着手可 |
| gBizINFO v2 | アクセストークン | 取得済み | v1は2026年9月終了のためv2実装が必須 |
| 不動産情報ライブラリ | APIキー | 審査中（目安5営業日） | Actor#3の公開ゲート |
| 国税庁 法人番号Web-API | アプリケーションID | 審査中（1〜1.5ヶ月） | **Actor#4の公開ゲート** |
| 法令API v2 | 不要 | — | ゲートなし |

## 3. 利用者・買い手

一次: 英語圏の開発者・データチーム（Apify Store経由、投資リサーチ／セールスインテリジェンス／PropTech／リーガルテック）。二次: AIエージェント（agentic実行）。三次: 国内ユーザー（特に#2補助金は入札Actorと同一買い手層で、クロスセルを要件に含む）。

## 4. 共通機能要件（FR-C）

- **FR-C1 二言語出力**: 全datasetアイテムはsnake_caseの英語フィールドを正とし、日本語原文を`*_ja`で併記する。
- **FR-C2 共通メタ**: 全アイテムに `source` / `source_url` / `retrieved_at`(ISO8601) / `attribution`(指定文言) / `schema_version` を付与する。
- **FR-C3 出典表記**: データ源ごとの指定文言（引継書§4に全文）をdatasetアイテムとREADMEの両方に必ず含める。
- **FR-C4 正直明記**: READMEに「Official API based — no scraping」を冒頭明記し、カバレッジ・更新頻度・非保証事項・**できないこと**を明記する。
- **FR-C5 エージェント対応**: 全Actorで `allowsAgenticUsers: true`。入力スキーマはフラットかつ必須項目最小。x402は有効化のみ（追加開発なし）。
- **FR-C6 無料枠**: 月間の無料利用枠を設ける（#1=20書類 / #2=200件 / #3=300件 / #4=100社 / #5=100条）。「試せるが業務には足りない」水準。
- **FR-C7 実行上限（コスト・レート防護）**: 1実行あたりの上限を設ける。#1: 一覧走査は31日分まで・書類取得500件まで／#3: 都道府県×年の組合せ12まで／#4: 1,000社まで／#5: 200条まで。超過指定はエラーではなく上限で打ち切り＋警告ログ。
- **FR-C8 部分失敗の許容**: 1レコードの失敗で実行全体を落とさない。失敗レコードは`_error`付きで出力し継続。実行自体の失敗条件は「認証エラー」または「失敗率50%超」。

## 5. Actor別機能要件

### FR-1 Actor#1: Japan Company Filings（EDINET）〔旗艦・最優先〕
- 入力: `date_from`/`date_to`、`doc_types[]`（既定: 120有価証券報告書・160半期報告書。訂正130は`include_amendments`で明示指定）、`edinet_codes[]`/`sec_codes[]`（任意）、`enrich`(bool)。
- 出力(basic): 書類メタ（docID, filer名 日/英, edinet_code, sec_code, 法人番号, doc_type, 期間, 提出日時）＋主要財務値（売上・営業利益・純利益・総資産・純資産・従業員数。単位正規化、和暦→ISO）。
- **v1スコープ制限**: XBRLフルパースは行わない。財務値はEDINETのCSV出力（type=5）から取得できる範囲に限定し、その旨をREADMEに明記する。
- 出力(enriched): 事業概要・主要リスク・セグメント構成の英文サマリ（各2〜3文）。固有名詞・数値は逐語照合（§6 N-9）を通過したもののみ。
- 正直明記: ファンド開示が混在すること（絞り込み方法を案内）／英文サマリは参考情報であり投資助言ではない／提出日ベースで決算期とずれる。

### FR-2 Actor#2: Japan Subsidies & Grants（gBizINFO補助金）
- 入力: `corporate_numbers[]` または `company_names[]`（番号解決は#4のコアを利用）、または `date_from/to`＋`ministry`（横断）。
- 出力(basic): 補助金名 日/英、府省(英)、金額(JPY)、日付、受給者（法人番号・名称 日/英）、`data_origin`（jGrants由来の識別）。
- **enriched層は設けない**（英語化は辞書＋ルール中心でbasicに内包）。
- 正直明記: 掲載は各府省の公開許諾分のみで悉皆ではない／2025年6月〜2026年1月にデータ凍結期間があった。
- 統廃合設計: KPI未達時は#4へ補助金フィールドとして吸収し単体廃止（§10）。

### FR-3 Actor#3: Japan Real Estate Transaction Prices（不動産情報ライブラリ）
- 入力: `year`(＋quarter)、`prefectures[]`/`cities[]`/`station`、`price_category`（取引価格/成約価格）、`property_types[]`。市区町村コードはActor内部で解決する。
- 出力(basic): APIが返す日英両フィールドをsnake_case正規化＋派生指標（㎡単価、取引時築年数）。**LLMを使用しない**（源データが日英併記のため）。
- 追加出力: 市区町村×四半期の集計dataset（中央値・件数）をオプションで出力（追加課金なし、basicに内包）。
- 正直明記: 取引価格（アンケートベース）と成約価格の性質差／悉皆でない／該当なしは0件として返る仕様。クレジット指定文言必須。

### FR-4 Actor#4: Japan Company Data Enrichment（法人番号＋gBizINFO）
- 入力: `corporate_numbers[]` または `company_names[]`、`fields[]`（取得ブロック選択）、`enrich`(bool)。
- 出力(basic): 法人番号、名称 日/英、住所 日/英、業種、資本金、従業員数、設立、URL、行政サマリ（補助金有無・調達有無・特許件数）。
- 名称英語の優先順位: gBizINFO登録英名 → ルール変換 → LLM翻字（`confidence`と`method`必須）。
- 出力(enriched): 事業概要の英文1行＋翻字のLLM検証。
- **公開ゲート**: 国税庁アプリケーションID到着（§2）。API接続部は柱3アドオンと共用（引継書§14）。
- 正直明記: gBizINFOのカバレッジは約400万法人・行政データは悉皆でない／翻字は機械生成。

### FR-5 Actor#5: Japan Laws & Regulations（法令API v2）〔最後発〕
- 入力: `law_query`（名称/法令番号/法令ID）、`articles[]`（条・項指定）、`as_of_date`（時点指定）、`translate`(bool)。
- 出力(basic): law_id、law_num、題名 日/英、条/項/号の構造化、公布・施行日、条文原文。
- 出力(translated): 条単位の英訳＋要約。「参考訳であり法的効力・助言ではない」disclaimerを全アイテムとREADMEに付す。
- **設計制約（要件）**: 全法令の順次一括取得は提供元の禁止事項のため実装しない。コーパス需要はREADMEで公式XML一括ダウンロードへ誘導する。
- 差別化の正直な位置づけ: 法務省の公式英訳DB（JLT）収録法令はJLT参照を推奨と明記し、本Actorの価値を「JLT未収録・最新改正のギャップ埋め」と宣言する。

## 6. 非機能要件（N-1〜N-9）

| # | 要件 |
|---|---|
| N-1 | データ源は公式APIのみ。直列実行・源別の既定リクエスト間隔（引継書§4）。429/403は指数バックオフ、3回で中断＋通知 |
| N-2 | LLM単価・為替の改定を検知したら原価を再計測し、価格見直しを起票する |
| N-3 | 源別の凍結fixtureに対するgolden回帰テストをCI必須とする。スキーマドリフト検知でCI fail |
| N-4 | 監視・通知: 実行失敗率>20%/日、スキーマドリフト、レート制限検知でオーナーに通知 |
| N-5 | 保守予算: 各Actor月1時間以内。2ヶ月連続超過で統廃合検討を自動起票 |
| N-6 | 扱うのは公表情報のみ。ユーザー識別はApify提供情報に限定。開示書類中の役員氏名等は公開情報として扱い、問い合わせ窓口をREADMEに明記 |
| N-7 | 出典表記の自動挿入（FR-C3の機構要件） |
| N-8 | `allowsAgenticUsers: true`、x402有効化（四半期$0なら追加対応なし。単月$200超で初めて拡張検討） |
| N-9 | LLM生成の品質規律: ①逐語照合（固有名詞・数値は原文一致検証。数値・固有名詞は照合失敗時null＋`verification_failed`、要約文はフラグのみ） ②原文に無い項目はnull（推測禁止） ③生成項目に`confidence`と`method`("api_native"/"rule"/"llm")必須 |

## 7. 課金要件（PPE）

レンタル課金は廃止済みのためpay-per-eventのみ。取り分80%。**enriched単価はマージン85%以上を下限条件**とし、原価はN-2で再計測する。

| イベント | 単価 | 対象 |
|---|---|---|
| actor-start | $0.02 | 全Actor |
| record-basic | #1 $0.005 / #2 $0.004 / #3 $0.003 / #4 $0.006 / #5 $0.004 | 全Actor |
| record-enriched | #1 $0.049（実測後 $0.049〜0.059で確定）/ #4 $0.019 | #1 #4 |
| article-translated | $0.029 | #5 |

## 8. 受入基準（Actor 1本の公開条件）
