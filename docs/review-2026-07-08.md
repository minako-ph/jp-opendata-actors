## F-1【重大・公開ブロッカー】APIキーがエラーメッセージ経由で利用者に漏洩する

**実証済みの事実**: `GovHttpClient`の`RateLimitAbortError`・`HttpStatusError`はメッセージに**リクエストURL全体**を含む。EDINETのキーはクエリ`Subscription-Key=`で送られるため、403/429のバックオフ枯渇時・HTTP 4xx/5xx時にキーがメッセージへ混入する（レビュー時に403スタブで実証: `url=...Subscription-Key=SECRET_KEY_12345`）。

**漏洩経路（いずれも利用者可視）**:
1. `run.ts`の`RunFailedError(String(error))` → `main.ts`の`Actor.fail(message)` → **実行ステータスメッセージ**
2. `deps.log.warning(...String(error))` → **Apify実行ログ**
3. `_error: String(error)` → **利用者のdataset**

**修正（実装箇所: `packages/gov-clients/src/http.ts`）**:
1. URLサニタイズ関数を追加し、**両エラークラスのコンストラクタ内部で**適用する（呼び出し側の付け忘れを構造的に防ぐ）:
   ```ts
   /** エラーメッセージ用にURLからクエリを除去する（キーはクエリで送られるため全除去が最も安全） */
   export function redactUrlForError(url: string): string {
     try {
       const u = new URL(url);
       return `${u.origin}${u.pathname}${u.search ? '?[query redacted]' : ''}`;
     } catch {
       return '[invalid url]';
     }
   }
   ```
   `RateLimitAbortError`と`HttpStatusError`のコンストラクタで`redactUrlForError(url)`を使ってメッセージを組み立てる。
2. **回帰テストを追加**（`packages/gov-clients/test/http.test.ts`）:
   - 403×バックオフ枯渇 → `RateLimitAbortError.message`に`Subscription-Key`とキー値が**含まれない**こと
   - 500応答 → `HttpStatusError.message`に同上
3. 将来のクライアント（法人番号はクエリ`id=`でアプリケーションIDを送る）も自動的に保護されることをコメントに明記。

## F-6【小】pnpm-workspace.yamlの`allowBuilds`キー

`allowBuilds`はpnpmの正式キーではない（esbuildはoptionalDependencies方式のため実害は出ていない）。正式な`onlyBuiltDependencies`へ置換:
```yaml
onlyBuiltDependencies:
  - esbuild
```
置換後に`pnpm install`が警告なく通ることを確認。

## F-7【運用・launch文書追記】組み込みイベントの二重課金防止を明示化

`docs/research/apify-ppe.md`が特定済みのとおり、組み込み`apify-default-dataset-item`はdataset書き込み毎に課金されるため、**`record-basic`と二重課金になる**。`docs/launch/edinet-filings.md`のチェックリスト#5（PPE設定）に明示追記: 「組み込み`apify-default-dataset-item`イベントは削除する（record-basicと二重課金になるため）」。
